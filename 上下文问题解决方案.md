# 上下文长度问题解决方案

## 问题描述

当测试设计节点较多时，会面临以下挑战：

1. **输入内容过长**: RS + TS + Mapping 可能超过 10K-50K 字符
2. **输出内容过长**: 50-100 个测试点可能超过 LLM 输出限制
3. **生成质量下降**: 内容过多导致 AI 注意力分散，遗漏测试点

---

## 解决方案：智能分块生成器

实现了 `SmartChunkedGenerator`，根据内容长度**自动选择最优生成策略**。

### 三种生成策略

| 策略 | 适用场景 | LLM 调用次数 | 生成时间 |
|------|---------|------------|---------|
| **一次生成** | 小型项目 (< 6K tokens) | 1 次 | 5-10s |
| **按分支生成** | 中型项目 (6K-20K tokens) | 2-5 次 | 15-30s |
| **按叶子生成** | 大型项目 (> 20K tokens) | 10-100 次 | 1-5min |

### 策略选择逻辑

```python
def _select_strategy(self, input_tokens, output_tokens):
    total_tokens = input_tokens + output_tokens
    
    if total_tokens < self.max_input_tokens * 0.8:
        return "full"  # 一次生成
    
    elif self.stats["level_1_nodes"] <= 5:
        return "by_branch"  # 按分支生成
    
    else:
        return "by_leaf"  # 按叶子生成
```


## 策略详解

### 策略 1: 一次生成 (full)

```
输入：RS + TS + Mapping + 完整模板
  ↓
AI 一次性生成完整测试设计
  ↓
输出：完整 JSON
```

**优点**:
- ✅ 调用次数少 (1 次)
- ✅ 整体一致性好
- ✅ 实现简单

**缺点**:
- ❌ 受上下文限制
- ❌ 质量可能不稳定

**适用**: < 20 个测试点的小型项目

---

### 策略 2: 按分支生成 (by_branch)

```
Level 1 分支: [L0-数据结果检查，L1-配置/调度检查]
  ↓
对每个分支分别生成:
  分支 1 Prompt + 局部模板 → AI → 分支 1 结果
  分支 2 Prompt + 局部模板 → AI → 分支 2 结果
  ↓
合并所有分支结果
```

**优点**:
- ✅ 避免上下文超限
- ✅ 质量较高
- ✅ 调用次数可控 (2-5 次)

**缺点**:
- ❌ 需要合并结果

**适用**: 20-50 个测试点的中型项目

---

### 策略 3: 按叶子生成 (by_leaf)

```
叶子节点：[10-100 个]
  ↓
对每个叶子节点分别生成:
  叶子 1 Prompt + 相关信息 → AI → 叶子 1 内容
  叶子 2 Prompt + 相关信息 → AI → 叶子 2 内容
  ...
  ↓
组装完整设计
```

**优点**:
- ✅ 质量最高 (专注每个点)
- ✅ 支持并行处理
- ✅ 不受内容长度限制

**缺点**:
- ❌ 调用次数多
- ❌ 成本较高

**适用**: > 50 个测试点的大型项目

## 上下文优化技巧

### 1. 相关内容提取

不是将所有 RS/TS/Mapping 都传给 AI，而是**根据测试点类型提取相关内容**：

```python
def _extract_relevant_info(self, leaf_path, rs, ts, mapping):
    if "主键" in leaf_path:
        # 只提取主键相关字段
        relevant_fields = [f for f in fields if f.is_primary_key]
    elif "度量" in leaf_path:
        # 只提取度量字段
        relevant_fields = [f for f in fields if f.is_measure]
    else:
        # 使用全部信息
        relevant_fields = fields
    
    return {"fields": relevant_fields, "mapping": ...}
```

### 2. 内容压缩

```python
def _compress_if_needed(self, text, max_length=2000):
    if len(text) <= max_length:
        return text
    return text[:max_length] + "... (内容已压缩)"
```

### 3. 预提取关键信息

避免重复解析，提前提取关键信息：

```python
# 预提取
target_table = self._extract_target_table(ts)
source_tables = self._extract_source_tables(ts)
fields = self._extract_fields(ts)

# 每个叶子节点复用
for leaf in leaf_nodes:
    relevant_info = self._extract_relevant_info(
        leaf.path, fields=fields, ...
    )
```

## 并行处理 (可选)

对于按叶子生成策略，支持并行处理：

```python
design = generator._generate_by_leaf_parallel(
    rs, ts, mapping,
    max_workers=5  # 5 个并发
)
```

**性能提升**:
- 串行：10 个叶子 × 5s = 50s
- 并行 (5 workers): 10/5 × 5s = 10s

## 配置建议

### 小型项目 (< 20 测试点)

```yaml
generator:
  strategy: auto
  max_input_tokens: 16000
```

### 中型项目 (20-50 测试点)

```yaml
generator:
  strategy: auto
  max_input_tokens: 8000
  preferred_branch_strategy: by_branch
```

### 大型项目 (> 50 测试点)

```yaml
generator:
  strategy: auto
  max_input_tokens: 8000
  parallel: true
  max_workers: 5
  batch_size: 10
```

## 成本估算

以通义千问为例：

| 策略 | 输入 Token | 输出 Token | 调用次数 | 成本 (元) |
|------|----------|----------|---------|---------|
| 一次生成 | 6000 | 1500 | 1 | ~0.01 |
| 按分支 (3 分支) | 2000×3 | 500×3 | 3 | ~0.02 |
| 按叶子 (10 节点) | 1000×10 | 150×10 | 10 | ~0.05 |

**结论**: 即使按叶子生成，成本也在可接受范围内。

## 最佳实践

1. **默认使用 auto 模式** - 让系统自动选择

2. **监控 LLM 调用次数** - 设置调用上限防止超支

3. **实现缓存机制** - 避免重复生成相同内容

4. **进度可视化** - 显示生成进度提升用户体验

5. **错误重试** - 对失败的节点自动重试

6. **日志记录** - 记录每次生成的输入输出，便于调试

## 总结

通过智能分块生成器，实现了：

✅ **自动选择最优策略** - 根据内容长度自适应

✅ **解决上下文超限** - 分块生成，避免 token 超限

✅ **保证生成质量** - 每个分块专注少量内容

✅ **支持并行处理** - 提升大型项目生成速度

✅ **成本可控** - 即使分块，成本也在可接受范围
